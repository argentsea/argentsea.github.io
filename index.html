Features
Sharding
Sharding allows you to separate your data onto multiple database servers. This is essential for load-sharing on high-volume systems, but it can also improve global performance by using regional data centers. ArgentSea was designed to support sharded data (although sharding is definitely not required). Multi-threaded queries can combine data across multiple shards, or return the first valid result. Shard-aware objects make working with unique shard records as simple as working with non-sharded data.
Mapping
By adding attributes to your model classes, you can automatically map a property to a parameter or resultset column. The Mapper maps nullable types to/from Db Null values (or alternatively maps Guid.Empty and NaN to Db Null). The Mapper supports Enums and automatically converts to/from their base types or names. Tha Mapper also supports complex child objects, including lists and 1:1 child objects. By using Expression Trees exclusively, the Mapper offers hand-coded performance without all the labor of actually coding the ADO.Net mappings (and no reflection overhead).
Configuration
ArgentSea moves configuration into the 21st century by abandoning connection strings. You still have access to all provider connection parameters (and you can create connection strings when you need them). By fully implementing the .NET core configuration architecture, you can store connection information data appropriately. For example, you could store your server name in an environment variable, the database name in a configuration value, and your passwords in a Key Vault or User secrets. This improves security and simplifies managing configuration and deployments across environments.
Resilience
ArgentSea automatically retries database commands that fail due to transient errors. You have per connection control over when, and how many, retries should be attempted. On persistent failures, “circuit breaking” means that failures don’t cause backlogged processes to overwhelm the request queue — this may keep a server alive, even if a data connection is down.
Performance
ArgentSea is written with performance in mind. It assumes the use of optimized stored procedures for all database activity. The mapping component uses Expression Trees, so it offers native-code performance for all mapped activity. Shard queries are multi-threaded, for simultaneous access across multiple shards. If you are looking only for one result, you don’t even have to wait for all the shard queries to finish.
Tooling
Robust, high-performance logging provides monitoring insight into the library behavior, performance, and operational problems. This includes the code that is generated by the mapper’s Expression Trees. Log to Application Insights, Amazon CloudWatch, Windows, or any .NET logging solution. ArgentSea also includes a separate desktop application that can further reduce coding labor by automatically creating read/insert/update/delete stored procedures from tables and also model classes (with attributes) from stored procedures.
Supported databases
• Sql Server 2017 and older
• Sql Server Express
• Azure Sql Database
• PostgreSQL database
• Amazon RDS for PostgreSQL
• Azure Database for PostgreSQL
• Google Cloud Sql for PostgreSQL
Status
  • The ArgentSea library is in beta phase. Object definitions could still change, but major changes are not expected.
  • All current tests are passing, but more tests are needed.
  • Documentation is nearing first release. QuickStart 2 is pending.
  • Desktop tooling is on hold.
  • NuGet package pipeline is awaiting a code signing certificate.
