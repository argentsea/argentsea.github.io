Features
Sharding
Sharding allows you to separate your data onto multiple database servers. This is essential for load-sharing on high-volume systems, but it can also improve global performance by using regional data centers. ArgentSea was designed to support sharded data (although sharding is definately not required). Multi-threaded queries can combine data across multiple shards, or return the first valid result. Shard-aware objects make working with unique shard records as simple as working with non-sharded data.
Mapping
By adding attributes to your model classes, you can automatically map a property to a parameter or resultset column. The Mapper maps nullable types to/from Db Null values (or alternatively maps Guid.Empty and NaN to Db Null). The Mapper supports Enums and automatically converts to/from their base types. Tha Mapper also supports complex child objects, including lists and 1:1 child objects. By using Expression Trees exclusively, the Mapper offers hand-coded performance without all the labor of actually coding the ADO.Net mappings (and no reflection overhead).
Configuration
ArgentSea moves configuration into the 21st century by abandoning connection strings. You still have access to all provider connection parameters (and can also create connection strings when you need them). By fully implementing the .NET core configuration architecture, you can store connection information data appropriately. For example, you could store your server name in an environment varible, the database name in a configuration value, and your passwords in a Key Vault or User secrets. This simplifies managing configuration and deployments in all environments and improves security.
Resilience
ArgentSea uses Polly to orchestrate automatic retries on transient errors. On persistant failures, Polly allow allows “circuit breaking” so that failures don’t cause backlogged processes to overwhelm the request queue. If ArgentSea is deployed in a globally distributed implementation, it also is smart enough to detect read failures that may be due to synchronization latency and automatically retry using the master connection.
Performance
ArgentSea is writen with performance in mind. It assumes the use of optimized stored procedures for all database activity. The mapping component uses Expression Trees, so it offers native-code performance for all mapped activity. Shard queries are multi-threaded, for simultaneous access across multiple shards. If you are looking only for one result, you don’t even have to wait for all the shard queries to finish.
Tooling
Robust, high-performance logging in ArgentSea allows support and troubleshooting teams to see rich process information. This includes the code that is generated by the mapper’s Expression Trees. Any implementation of ILogger can be used. ArgentSea allow includes a separate desktop application that can further reduced coding labor by automatically creating read/insert/update/delete stored procedures from tables and also model classes (with attributes) from stored procedures.

Paradigm Shifty
Tight Coupling
Although it might sound cozy, “tight coupling” isn’t a good thing in software design. It happens when one system’s integration with another system depends on the internal implementation of the other system. Usually the result of haphazard design, it makes it nearly impossible to switch to a different provider.

Like, say, when accessing your database layer depends upon how tables and columns are implemented.

Loose Coupling
“Loosely coupled” systems have well-defined interfaces. You can change the implementation as long as you maintain the interface contract.

These systems are more robust, testable, and manageable — and those are good things.

For example, using stored procedures allows you to change the underlying database structures, but as long as the same results are returned you will not break the application.

Stored procedures
Stored procedures enable loose coupling between the app domain and the data domain. They generally perform better and offer better security too.

ArgentSea was built to work with stored procedures.

Do your data-domain work in SQL and your application work in .NET. That is how you build a maintainable high-performance application. (If you prefer to work in C# and have your ORM handle data for you automagically, there are several other great options for you).
